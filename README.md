# 【Git研修】タイムマシンを使ってみよう！GitHub Desktopで学ぶバージョン管理の第一歩 🚀

この研修では、ファイルの「変更履歴」を管理できる**Git**というシステムを、直感的に操作できる**GitHub Desktop**というアプリを使って学びます。

## なぜ「バージョン管理」が必要なの？🤔

**Git**とは、ファイルの「変更履歴」をきちんと記録してくれる、**タイムマシンのようなシステム**です。

- **🕰️ いつでも好きな時点に戻れる**：「3日前の状態に戻したい」が簡単にできる。
- **✍️ 誰が何をどう変更したか分かる**：変更履歴がすべて記録されるので、後から見返すことができる。
- **🤝 複数人で安全に作業できる**：他の人の作業を誤って上書きしてしまう事故を防ぐ。

## 1. 準備をしよう（セットアップ）

### GitHub Desktopのインストール

1. [GitHub Desktopの公式サイト](https://desktop.github.com/)にアクセス。
2. 「Download for macOS」ボタンをクリックし、インストーラーをダウンロード。
3. ダウンロードしたzipファイルを解凍し、GitHub Desktop.appをアプリケーションフォルダに移動する。
4. GitHub Desktopを起動し、TOKIUMのGitHubアカウントでログイン。

## 2. 【ワーク】キャラクターのプロンプトを作ってみよう！

ここからが本番です！汎用的なAIキャラクターの話し方を定義した「Prompt-template.txt」を元に、あなただけのオリジナルキャラクターのプロンプトを作成してみましょう。

### Step 1: 開発の拠点（リポジトリ）を自分のPCにコピーする（クローン）

まず、研修用のプロンプトが保管されているGitHub上の倉庫（リポジトリ）を、作業するために自分のPCに丸ごとコピーしてきます。この作業を**クローン**と呼びます。

1. GitHub Desktopを開き、メニューバーから File > Clone Repository... を選択する。
2. URLタブを選択し、GitHubリポジトリとして下記のURLを貼り付け。

```
https://github.com/TOKIUM/git-training-for-ai-agent-interns.git
```

3. Local Pathで、PC上のどこに保存するかを選択する。（どこでも大丈夫ですが、分からなければ「書類」フォルダなどがおすすめです）
4. Cloneボタンをクリック。

### Step 2: 自分専用の作業スペースを作る（ブランチ作成）

元のファイルを直接編集してしまうと、もし失敗した時に元に戻せなくなったり、他の人の作業に影響が出たりする可能性があります。

そこで、安全に作業するために、元のファイルの自分専用のコピー（ブランチ）を作成します。

1. GitHub Desktopの上部にある Current Branch が main になっていることを確認し、その隣の New Branch ボタンをクリック。
2. ブランチ名を入力する。他の人と被らないように「自分の名前/好きなキャラクター名」などで作成。（例: yamada/cat-character）
   - **💡Point:** ブランチ名に日本語は使わず、半角英数字とハイフン - を使いましょう。
3. Create Branch ボタンをクリック。

### Step 3: プロンプトを編集して、キャラクターに魂を吹き込む（ファイル編集）

いよいよプロンプトを編集します。自分の好きなアニメやゲームのキャラクター、あるいは想像上の人物になりきって、話し方や性格を定義してみましょう。

1. **プロンプトファイルを編集する**
   1. GitHub Desktopで、Show in Finder ボタンをクリックし、PCにコピーしたフォルダを開く。
   2. 中にある Prompt-template.txt を見つけ、エディタで開く。
   3. キャラクターの一人称、口癖、性格などを自由に編集・追記。
2. **Geminiで試してみる**
   1. [Gemini](https://gemini.google.com/) にアクセスする。
   2. プロンプトをGeminiに送信。
   3. そのキャラクターに話しかけたい内容を送信し、どんな返事が返ってくるか確認。（例: 「今日の天気はどう？」、「自己紹介してください」など）
3. **結果を見て、さらに改良する**
   - もしイメージと違ったら、プロンプトファイルに戻り指示の出し方を調整する。
   - **「編集 → Geminiで試す → 改良」** のサイクルを完成まで繰り返す。

### Step 4: 作業内容をセーブする（コミット）

ファイルの編集が終わったら、「どこをどう変更したか」というメモと一緒に、変更内容を記録します。この作業を**コミット**と呼びます。ゲームでセーブポイントを作る感覚に似ています。

1. GitHub Desktopの画面に戻ると、左側のChangesタブに変更したファイルが表示されている。
2. 画面左下にあるSummary（概要）の欄に、**何を変更したか**が簡潔にわかるタイトルを入力する。（例: 猫キャラクターの口調を追加）
3. （任意）Description（詳細）の欄に、より詳しい説明を書くこともできる。
4. Commit to （作成したブランチ名） ボタンをクリック。

### Step 5: セーブデータをみんなの倉庫に送る（プッシュ）

コミット（セーブ）は、まだ自分のPC上に記録されているだけです。この記録を、インターネット上の倉庫（GitHub）にも反映させ、他の人からも見られるようにします。この作業を**プッシュ**と呼びます。

1. GitHub Desktopの上部にある Publish branch または Push origin という青いボタンをクリック。

### Step 6: 「この変更どうかな？」とレビューをお願いする（プルリクエスト作成）

あなたが加えた変更を他の人に確認してもらうための依頼を作成します。これを**プルリクエスト（Pull Request）**と呼びます。

チーム開発では、このプルリクエストを通じて、他のメンバーに変更内容をレビューしてもらい、問題がなければ元のファイルに統合（マージ）するという流れで進みます。

**ただし、今回はレビュー依頼を出すところまでをゴールとし、統合（マージ）は行いません。**

1. プッシュが完了すると、GitHub Desktopの上部に Create Pull Request というボタンが表示されるので、これをクリック。
2. 自動的にブラウザが開き、GitHubのプルリクエスト作成画面が表示される。
3. タイトルや説明を記述する欄があるが、今回はそのまま。
4. 画面右側の Create Pull Request ボタンをクリック。

## 3. まとめ

### 用語集

| 用語 | 説明 | 比喩 |
| --- | --- | --- |
| **リポジトリ** | ファイルやフォルダ、そしてそれらの変更履歴を保管しておく場所。 | インターネット上の倉庫 |
| **クローン** | リポジトリを自分のPCに丸ごと複製すること。 | 倉庫から品物を一式借りてくる |
| **ブランチ** | 変更履歴の流れを分岐させて記録していくためのもの。安全に作業するための自分専用のスペース。 | 設計図のコピー |
| **コミット** | ファイルの追加や変更を、リポジトリに記録（セーブ）すること。 | セーブポイントを作る |
| **プッシュ** | 自分のPCで行ったコミットを、GitHub上のリポジトリに反映させること。 | セーブデータをサーバーに送る |
| **プルリクエスト** | 自分の変更内容を他の人に通知し、レビューや統合（マージ）を依頼する機能。 | 「この変更どうですか？」という提案書 |
| **マージ** | あるブランチで行った変更を、別のブランチに取り込み、合体させること。 | 自分がコピーした設計図に書き加えた変更を、元の設計図に反映させること。 |
| **コンフリクト** | 複数の人が同じファイルの**同じ箇所**を別々に変更してしまい、Gitが自動でマージできなくなった状態。「どちらの変更が正しいですか？」とGitに聞かれている状態。 | 複数人が同じレポートの同じ行を、違う内容に書き換えて提出した状態。手動で調整が必要になる。 |

### 困ったときは（FAQ）

- **Q. 間違えてコミットしてしまいました。取り消せますか？**
  - A. はい、取り消せます。GitHub Desktopでは、Historyタブからコミットを右クリックし、Revert Changes in Commitを選択することで、変更を打ち消す新しいコミットを作成できます。分からなければ、すぐに研修担当者に相談しましょう！

1. **クローン** - リポジトリを自分のPCにコピー
2. **ブランチ作成** - 自分専用の作業スペースを作成
3. **ファイル編集** - Prompt-template.txtを編集してキャラクターを作成
4. **コミット** - 変更内容を記録
5. **プッシュ** - 変更をGitHubに送信
6. **プルリクエスト作成** - レビュー依頼を作成
